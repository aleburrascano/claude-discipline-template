# ADR-0001: Hexagonal Architecture (Ports & Adapters)

- **Status:** Accepted
- **Date:** {{DATE}}
- **Deciders:** {{DECIDERS}}
- **Context tags:** [arch, foundational]

## Context

{{PROJECT_NAME}} needs a backend architecture that:

- Keeps business logic testable without spinning up infrastructure (DB, HTTP, queues).
- Allows changing technology choices (database vendor, HTTP framework, message broker) without rippling through business code.
- Provides clear, enforceable boundaries so a solo dev or small team doesn't drift into mud-ball coupling over months/years.
- Has mature ecosystem references (vault: Cockburn, DDD-Hexagon, Clean Architecture).

The dev's working assumption is that this is a long-lived project where correctness + maintainability matter more than initial velocity.

## Decision

Adopt **Hexagonal Architecture** (also called Ports & Adapters). Layer structure:

- `domain/` — pure business model. No framework, no I/O.
- `application/` — use cases + port interfaces. Depends on `domain/` + stdlib.
- `adapters/inbound/` — HTTP routers, CLI, message consumers. Translate external requests into use case calls.
- `adapters/outbound/` — DB repositories, external clients, publishers. Implement ports defined in `application/`.
- `platform/` — DI container, config, logging. Wires everything.

Dependency rule: **inward only**. `adapters → application → domain`. `platform → everything`.

Per-bounded-context organization within each layer (e.g., `domain/catalog/`, `application/catalog/`, `adapters/outbound/persistence/catalog/`).

## Alternatives considered

| Alternative | Why not |
|---|---|
| **Layered (Controller → Service → Repository) without ports** | Couples application to specific repository implementation; testing requires DB. Hexagonal gives the port seam for free. |
| **Clean Architecture (formalized)** | Effectively the same shape as hexagonal at this scale; the explicit "use cases" + "interface adapters" layers add ceremony without benefit for a small project. Easy to graduate later if it ever matters. |
| **Onion** | Same family as hexagonal; we pick hexagonal naming because the vault note is more thorough and the port/adapter vocabulary maps cleanly to most DI frameworks. |
| **Vertical-slice (no layers)** | Great for greenfield CRUD-y apps; harder to keep clean as cross-feature concerns emerge. Combinable with hexagonal at the per-feature level. |
| **MVC** | Too thin for non-trivial business logic; muddles model-vs-domain. |
| **Flat / no architecture** | Works for prototypes; collapses into mud at scale. |

## Consequences

### What becomes easier
- Domain + application testable without DB or framework (in-memory adapters).
- Swapping infrastructure choices is contained to the adapter; domain/application untouched.
- Reviews become mechanical: did the change cross a boundary? The `.claude/rules/{domain,application,adapters}-layer.md` files codify the answer.

### What becomes harder
- More indirection than a flat MVC for trivial CRUD endpoints. Worth it on anything non-trivial.
- Discipline to maintain the port/adapter distinction. Mitigated by hooks + `architecture-reviewer` subagent.
- Constructor injection boilerplate. Mitigated by centralizing wiring in `platform/container.py`.

### What we're committing to (and the cost to reverse)
- This layering is committed for the lifetime of the backend. Relaxing it later means rewriting around a different shape, which is much worse than tightening it as we go.
