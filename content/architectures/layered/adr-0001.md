# ADR-0001: Layered Architecture (Controller → Service → Repository)

- **Status:** Accepted
- **Date:** {{DATE}}
- **Deciders:** {{DECIDERS}}
- **Context tags:** [arch, foundational]

## Context

{{PROJECT_NAME}} adopts a classic n-tier layered architecture: Presentation → Application/Service → Persistence. Suitable for CRUD-heavy applications where the domain logic isn't deep enough to justify hexagonal's port/adapter ceremony.

## Decision

Three primary layers:

- `<your-pkg>/presentation/` (or `routes/`, `controllers/`) — HTTP handlers. Thin: parse request → call service → serialize response.
- `<your-pkg>/services/` (or `application/`) — business logic. Orchestrates between persistence and external services. Where transactions begin and end.
- `<your-pkg>/persistence/` (or `repository/`, `data/`) — database access. Repository pattern recommended even here (see vault).
- `<your-pkg>/models/` — shared data types (Pydantic, dataclasses, TypeScript interfaces).

Dependency rule: **downward only**. Presentation depends on services; services depend on persistence; persistence doesn't depend on anything above it.

## Alternatives considered

See [vault: wiki/concepts/Hexagonal Architecture.md] vs [vault: wiki/concepts/Layered Architecture.md] comparison. Hexagonal trades complexity for testability + swap-friendliness; layered trades testability for simplicity. For CRUD-heavy or short-lived projects, layered is the right call.

## Consequences

### What becomes easier
- Lower ceremony than hexagonal. Faster to ship CRUD endpoints.
- Familiar to most developers; less onboarding cost.

### What becomes harder
- Testing services often requires a database (or extensive mocking).
- Swapping infrastructure (e.g., DB vendor) ripples through services.
- Boundaries are convention, not enforced by separate ports — vigilance required.

### What we're committing to
- Reversing to hexagonal later requires extracting ports + introducing the application layer cleanly. Doable but not free.
